由为宁美魔方 mini 主机配置 PXE 网络启动和自动安装，看 PXE 网络启动、Linux 内核编译、autoinstall、ISO 启动镜像打包和 Linux 启动原理

1. 起因   test

1.1. 因为需要为没有网口的宁美魔方 mini 主机，开启 PXE 网络启动和 Ubuntu 系统自动安装工作流

以方便结合 Ubuntu 20.04 LTS Server 最新的 autoinstall 功能全自动批量安装系统。虽然该主机的 UEFI 有 PXE 网络启动选项，但因为苦于没有以太网接口，需要外接 USB 转以太网口转换器才可以连接网线。于是就有了一段漫长的踩坑旅程。

1.2. 所以一路探索挖了包括 PXE 搭建、Linux 内核编译、ISO 镜像打包和 Linux 启动等多个坑

事后追溯，实现最终目标正确路径如下：

  • 使用 USB 转以太网口转换器连接网线，解决有线连接问题
  • 编译 iPXE 制作成镜像 U 盘启动，以解决默认 PXE 无 USB 转以 太网驱动问题
  • 搭建 PXE 服务，并改写 grub 配置问题，以实现自动安装
  • 往 Ubuntu 20.04 LTS Server 默认内核添加USB转以太网驱动，以实现 autoinstall 自动安装 Ubuntu 过程中可以连接有线网络
  • 最终结果，直接用一批修改过启动镜像的 TF 卡自动启动代替

现将过程记录如下。

2. USB 转以太网转换器

拿到的宁美魔方非常小巧，外部接口有两个 USB 3.0，一个用作电源接口的 USB-TypeC，一个 HDMI，一个 TF 卡槽和一个 3.5mm 耳机孔，没有提供 RJ-45 以太网接口。

而 PXE 启动需要一个网络连接，首先就是有线网络连接。（无线网络事后来看通过 iPXE 理论上可以支持，没有过多尝试）

首先尝试官宣支持 PXE 的 Dell Adapter - USB 3.0 to Ethernet PXE Boot 失败，原因可能是需要 DELL UEFI 的配合。

然后准备某宝随机挑选了一款 USB 转以太网转换器，希望可以通过改造或替换添加过相关驱动的 UEFI 以达到目的，需要改造自然首选开源，选定的开源 UEFI 方案是 coreboot，经验证它不支持对手上的设备采取较简单的/在线烧录/方式，而复杂的方式则是需要拆机用烧录器实现，因此结论是 UEFI 替换过程及其复杂，不具有操作性。

3. 编译 iPXE 启动镜像

除了动 UEFI 的心思，接下来的思路是将 PXE 动能单拎出来，先启动到一个第三方 PXE 上，通过这层跳板使用 PXE 服务。

首先想到的是 Clonezilla，它有一个 iPXE 启动选项，通过 U 盘启动到 Clonezilla 镜像，选择 iPXE 启动，进而实现 PXE 服务。想法很美好，尝试之后结论是 Clonezilla 的 iPXE 没法驱动 USB转以太网转换器。

转而直接研究 iPXE，通过其未被 Google 索引的论坛找到了为 USB 转以太网转换器特别准备的编译选项。参考 iPXE 官网文档，具体过程如下： 

git clone git://git.ipxe.org/ipxe.git
cd ipxe/src
make bin-x86_64-efi/ecm.usb
sudo dd if=bin-x86_64-efi/ecm.usb of=/dev/sdc

其中 bin-x86_64-efi/ecm.usb 三部分意义分别是平台、驱动、类型。

这样，一个专门启动 iPXE 的跳板镜像就完成了。它使用 GPT 分区表（因为默认 UEFI 识别 GPT 分区表，这样就省去了 MBR 分区表需要进 UEFI 开启 CSM 兼容模式的麻烦）。

4. 部署 PXE 服务

前面的步骤是为了给主机提供一个可以使用的 PXE 客户端，接下来的任务是创建一个 PXE 服务。PXE 服务需要 DHCP、TFTP 和一个外部文件服务（这里使用 HTTP ）。

4.2. 带额外字段的 DHCP 服务，TFTP 服务

DHCP 和 TFTP 是分开的两个服务，这里选用 dnsmasq 一个小型 DNS 服务附带 DHCP 和 TFTP 功能，可完美解决这里的需求。

dnsmasq 默认两个配置文件 /etc/dnsmasq.conf 内容如下： 

felix@master:~$ grep '^[^#]'  /etc/dnsmasq.conf
port=53 # dns服务端口
domain-needed
dhcp-option=option:router,192.168.33.222        # dhcp服务器
enable-tftp     # 开启tftp服务
tftp-root=/var/ftpd     # 配置tftp内容位置
log-dhcp
dhcp-range=192.168.33.100,192.168.33.199,2h     # dhcp分配地址范围
dhcp-boot=grubnetx64.efi        # pxe网络启动的efi启动文件
dhcp-host=84:39:be:9c:0b:f3,192.168.33.222      # 绑定某服务器ip地址

解决 dnsmasq 和 systemd-resolved.service 的冲突问题，简单关闭掉 systemd-resolved.service？忘了。

现在看 /var/ftpd 中文件，这是 PXE 客户端读取并执行的文件， 

felix@master:/var/ftpd$ ls
casper  grub  grubnetx64.efi  ldlinux.c32  pxelinux.0  pxelinux.cfg

grubnetx64.efi 是要执行的 efi程序，配套 grub 文件夹中的配置文件 =/var/ftpd/grub/grub.cfg=， 

if loadfont /boot/grub/font.pf2 ; then
    set gfxmode=auto
    insmod efi_gop
    insmod efi_uga
    insmod gfxterm
    terminal_output gfxterm
fi

set menu_color_normal=white/black
set menu_color_highlight=black/light-gray

set timeout=5
menuentry "Install Ubuntu Server" {
    set gfxpayload=keep
    linux       /casper/vmlinuz   quiet  root=/dev/ram0 ramdisk_size=1500000 ip=dhcp url=http://192.168.33.222/ubuntu-20.04.1-live-server-amd64.iso ---
    initrd      /casper/initrd
}
menuentry "Install Ubuntu Server (safe graphics)" {
    set gfxpayload=keep
    linux       /casper/vmlinuz   quiet  nomodeset root=/dev/ram0 ramdisk_size=1500000 ip=dhcp url=https://mirrors.tuna.tsinghua.edu.cn/ubuntu-releases/focal/ubuntu-20.04.1-live-server-amd64.iso ---
    initrd      /casper/initrd
}
grub_platform
if [ "$grub_platform" = "efi" ]; then
    menuentry 'Boot from next volume' {
        exit
    }
    menuentry 'UEFI Firmware Settings' {
        fwsetup
    }
fi

其中， 

linux   /casper/vmlinuz   quiet  root=/dev/ram0 ramdisk_size=1500000 ip=dhcp url=http://192.168.33.222/ubuntu-20.04.1-live-server-amd64.iso ---
initrd  /casper/initrd

指导 tftp 下的内核启动 192.168.33.222 的 HTTP 服务中的 .iso 镜像，Initial ramdisk 指定为 tftp 服务下的 /casper/initrd，至此 PXE 服务畅通。

5. 编译 Ubuntu 内核，并打包 iso 启动镜像

接下来的方向应该是读取到 .iso 文件，开启一段愉快的网络安装体验，前提是上一步的 /casper/vmlinuz 有 USB 转以太网驱动的话。很遗憾，没有。从 Ubuntu Server 安装镜像提取到的 /casper/vmlinuz 不带此驱动，从对应桌面版 Ubuntu 安装镜像提取到的虽然有此驱动，但是无法使用 autoinstall 功能。

接下来的工作只需要为 /casper/vmlinuz 编译一个带 USB 转以太网驱动的内核以替代前者，参考这里， 修复 _apt 权限问题， 

sudo chown -Rv _apt:root /var/cache/apt/archives/partial/
sudo chmod -Rv 700 /var/cache/apt/archives/partial/

使用 apt 下载 Linux 内核代码当前目录， 

sudo apt source linux-source-5.4.0

编译作业待验证

下一步是改造官方的 .iso 镜像，以为安装内核植入 autoinstall 参数，以实现全自动安装过程。基本思路是首先拆解 .iso 镜像，编辑对应文件，更新 md5 文件，打包回 .iso 自定义镜像。

6. 最终，使用修改过的 Ubuntu 启动镜像 TF 卡插卡直接启动

实际中放弃了全自动安装过程，转而直接改造 .iso 镜像的启动参数，注入 auto install 自动启动，然后把这个镜像烧录进 TF 卡不也可以实现半自动化的安装吗！这时候发现一个可以无脑定制 Ubuntu 镜像的工具 Cubic，其讲 Ubuntu 镜像的结构直观得展示在出来，并可以随意改造。

很自然，用 Cubic 直接改造镜像加入自动安装机制。不同的是放弃 PXE 启动，转而找了一批 TF 卡，用改造后的启动镜像，实现半自动安装。

改造的地方包括在根目录放入一个 autoinstall 文件夹，参考 0.5.1 的内容；修改 /boot/grub/grub.cfg 的内核启动参数为， 

linux /casper/vmliuz boot=casper quiet autoinstall "ds=noclouds;s=/cdrom/autoinstall/" ---
initrd /casper/initrd.lz

casper 参数意为启动 live system，这里 autoinstall 用了 cloud-init 中的 noclouds。事实上 cloud-init 已经是云服务中系统初始化的标准，其他 Linux 发行版私以为也有类似形式的功能。
